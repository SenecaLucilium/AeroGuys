## 2. Постановка задачи

### 2.1 Проблема

Глобальные данные о воздушном трафике общедоступны через OpenSky Network, однако работа с ними требует знания API, программирования и специализированных инструментов. Отсутствуют готовые аналитические платформы, позволяющие без технических знаний:

- получать актуальные данные о рейсах,
- строить агрегированную статистику по аэропортам и воздушным судам,
- визуализировать маршруты и тенденции в удобном интерфейсе.

### 2.2 Цель проекта

Создать веб-приложение, которое:

1. Автоматически собирает данные о воздушном трафике из OpenSky Network.
2. Хранит исторические данные в реляционной СУБД.
3. Предоставляет аналитические дашборды с визуализацией через браузер.
4. Поддерживает экспорт данных для внешнего анализа.

### 2.3 Целевая аудитория

- Студенты и исследователи, изучающие авиационный трафик
- Специалисты по анализу данных (аналитики, дата-инженеры)
- Разработчики, желающие встроить авиационную аналитику в свои системы

## 4. Нефункциональные требования

| Категория         | Требование                                                           |
|-------------------|----------------------------------------------------------------------|
| **Производительность** | API-запросы должны выполняться менее чем за 2 с на типовых объёмах данных |
| **Масштабируемость**   | Хранилище должно поддерживать миллионы строк без деградации производительности (индексы) |
| **Доступность**        | Система должна запускаться одной командой `docker compose up`    |
| **Безопасность**       | Учётные данные передаются через переменные окружения, не хранятся в коде |
| **Переносимость**      | Полная контейнеризация; работает на любом хосте с Docker         |
| **Расширяемость**      | Добавление нового эндпоинта не требует изменений в слоях БД или фронтенда |

## 7. Описание модулей

### 7.1 `Analysis/queries.py` — DatabaseQueries

Центральный класс бизнес-логики. Содержит все аналитические SQL-запросы, сгруппированные по доменам. Принимает `DatabaseManager` через конструктор.

**Основные группы методов:**

| Группа            | Методы                                                                    |
|-------------------|---------------------------------------------------------------------------|
| Реальное время    | `get_current_positions`, `get_fastest_aircraft`, `get_highest_aircraft`   |
| Ртзгруженность    | `get_airport_busyness`, `get_city_busyness`                               |
| Аэропорты         | `get_airport_stats`, `get_airport_peak_hours`, `get_airport_destinations` |
| Воздушные суда    | `get_aircraft_by_callsign`, `get_snapshot_stats`, `get_flight_phases`     |
| Маршруты          | `get_popular_routes`, `get_route_efficiency`, `get_duration_distribution` |
| Авиакомпании      | `get_top_airlines_by_flight_count`                                        |
| Экспорт           | `export_flights_to_csv`, `export_analytics_zip`                           |

### 7.2 `OpenskyAPI/client.py` — OpenskyClient

HTTP-клиент для работы с REST API OpenSky Network.

**Основные возможности:**
- OAuth2-аутентификация через `TokenManager`
- Автоматические повторные попытки на ошибки 5xx (до 3 раз с backoff)
- Отслеживание лимитов API (`X-Rate-Limit-Remaining`)
- Ограничения интервалов запросов (не более 2 часов истории полётов, не более 30 дней для треков)

**Основные методы:**

| Метод                  | OpenSky эндпоинт       | Описание                                |
|------------------------|------------------------|-----------------------------------------|
| `get_states(bbox)`     | `GET /states/all`      | Текущие позиции всех ВС (или в bbox)    |
| `get_flights(begin, end)` | `GET /flights/all`  | Рейсы за временной интервал            |
| `get_track(icao24, time)` | `GET /tracks/all`   | Трек конкретного борта                 |

### 7.3 `parsing/data_collector.py` — DataCollector

Оркестрирует один цикл сбора данных:
1. Вызывает `OpenskyClient.get_states()` → сохраняет через `DataStorage`
2. Вызывает `OpenskyClient.get_flights()` → сохраняет через `DataStorage`
3. (Опционально) Вызывает `OpenskyClient.get_track()` для списка бортов

Обрабатывает `RateLimitError` (ждёт требуемое время) и `OpenskyAPIError`.

## 9. Развёртывание

### 9.1 Продакшн-развёртывание (Docker)

```bash
# Клонирование
git clone <url> && cd AeroGuys

# Настройка окружения
cp .env.example .env
# Заполнить .env: OPENSKY_CLIENT_ID, OPENSKY_CLIENT_SECRET

# Запуск
docker compose up --build -d

# Проверка
docker compose ps
curl http://localhost:8000/api/init/status
```

### 9.2 Остановка и очистка

```bash
# Остановить контейнеры
docker compose down

# Остановить и удалить данные БД
docker compose down -v
```

### 9.3 Переменные окружения

Создайте файл `.env` в корне проекта:

```env
# OpenSky Network API
OPENSKY_CLIENT_ID=your_client_id
OPENSKY_CLIENT_SECRET=your_client_secret

# PostgreSQL (используются бэкендом в Docker-сети)
DB_HOST=postgres
DB_PORT=5432
DB_NAME=aeroguys_db
DB_USER=aeroguys_user
DB_PASSWORD=aeroguys_password
```

## 10. Ограничения и известные особенности

| Ограничение                              | Причина                                   | Эффект                              |
|------------------------------------------|-------------------------------------------|-------------------------------------|
| Максимум 2 часа истории при сборе рейсов | Лимит OpenSky Network API                 | Для длинных периодов нужна регулярная загрузка |
| Загрузка треков ограничена 30 днями      | Лимит OpenSky Network API                 | Старые треки недоступны             |
| CORS разрешён только для localhost:3000  | Настройка разработки                      | В продакшне нужно обновить список источников |
| Аэропорты без ICAO-кода в справочнике    | Лogi нет явного FK flights → airports     | Рейсы через неизвестный аэропорт сохраняются, но без гео-информации |
| Ручной запуск сбора данных               | Нет встроенного планировщика (cron)       | Для периодического сбора нужен внешний cron или ручной запуск |
